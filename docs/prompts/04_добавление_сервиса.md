# 04 — Добавление сервиса

Используй этот промт, когда нужно добавить новый контейнер или фоновый процесс в Docker Compose проекта Whisper STT.

---

# Промт

Добавь новый сервис в проект Whisper STT sidecar:

```text
<ОПИСАНИЕ_СЕРВИСА>
```

---

# Контекст проекта

Текущий `docker-compose.yml`:

| Сервис | Назначение |
|--------|------------|
| `redis` | Очередь и хранение jobs |
| `whisper-stt` | FastAPI (`python -m app`) |
| `whisper-worker` | GPU worker (`python -m app.worker`) |

Оба Python-сервиса — **один образ**, разные `command` и `WHISPER_PROCESS_ROLE`.

---

# Критическое правило

```text
HTTP API must never block on GPU transcription.
```

Новый сервис не должен:

* выполнять STT вместо worker без явного обоснования;
* дублировать очередь Redis;
* ломать `REDIS_URL` для api/worker;
* использовать `localhost` между контейнерами.

---

# Перед реализацией

Оцени:

* можно ли реализовать в существующем `whisper-stt` или `whisper-worker`;
* нужен ли отдельный контейнер (второй worker на другом GPU, sidecar для метрик и т.д.);
* не создаёт ли сервис лишнюю сложность.

Если отдельный сервис не нужен — укажи это явно.

---

# Типичные допустимые сервисы

* второй **whisper-worker** (другой GPU / `NVIDIA_VISIBLE_DEVICES`);
* **redis** replica/sentinel (вне scope MVP);
* экспортер метрик / health proxy;
* отдельный **download/init** job (one-shot, не в compose permanently).

---

# Обычно НЕ нужно

* отдельный API для STT (уже есть whisper-stt);
* MySQL / PostgreSQL (проект без БД);
* Celery / RQ (используется native Redis LIST);
* React frontend (проект — API sidecar).

---

# Docker Rules

* сеть `whisper-net`;
* service names для inter-container URLs;
* env из `.env`, без hardcoded secrets;
* те же volumes для models/logs/config, если сервис транскрибирует;
* `restart: unless-stopped`;
* для GPU — `runtime: nvidia`.

При добавлении worker-клона:

* тот же образ и `command: ["python3", "-m", "app.worker"]`;
* общий `REDIS_URL` и `whisper:queue`;
* учесть `WHISPER_MAX_CONCURRENT_JOBS` на каждый worker.

---

# Healthcheck

* API: `GET /health` на порту 19900;
* worker: опционально проверка `whisper:worker:alive` в Redis (отдельный health sidecar редко нужен);
* redis: `redis-cli ping`.

---

# Logging

Worker и API пишут в том `/logs`. Новый сервис — либо тот же том, либо stdout для `docker compose logs`.
