# Контекст проекта

**Whisper STT sidecar** — GPU-сервис транскрипции с async queue.

```text
Client
  ↓ POST /transcribe (202)
whisper-stt (FastAPI, no GPU model)
  ↓ Redis: whisper:queue, whisper:job:*, whisper:dedup:*
whisper-worker
  ↓ download → ffmpeg → faster-whisper [→ pyannote]
  ↓ completed result
Client ← GET /jobs/by-dedup/{dedup_key}
```

Compose: `redis`, `whisper-stt`, `whisper-worker`.

Документация: `docs/architecture/`.

---

# Критическое правило

```text
HTTP API must never block on GPU transcription.
```

Оценивай все решения с точки зрения:

* isolation API vs GPU;
* надёжности очереди;
* recovery после сбоев;
* предсказуемости для интеграций (n8n, dedup_key).

---

# Главная задача

Проанализировать:

* текущую архитектуру и её границы;
* scalability (throughput jobs, GPU, Redis);
* bottlenecks (один worker, MAX_CONCURRENT_JOBS, VRAM);
* надёжность (orphan, stale, queue desync);
* maintainability модулей `app/`;
* готовность к production.

---

# Backend Architecture

* разделение `main.py` / `worker.py` / `transcribe_pipeline.py`;
* coupling между job_store и API;
* ThreadPoolExecutor vs asyncio;
* singleton models (whisper, pyannote);
* error propagation job → client.

Найди: layer violations, duplicated logic, tight coupling, hidden SPOF.

---

# Redis & Queue Architecture

* LIST queue vs streams (trade-offs);
* dedup index и lock;
* idempotency POST /transcribe;
* multi-worker scaling (несколько whisper-worker на одной queue);
* persistence (AOF);
* memory growth job records.

---

# Worker & GPU Architecture

* семафор и waiting_gpu;
* pyannote + whisper on same CUDA;
* model load time at startup;
* VRAM при turbo + diarization;
* horizontal scaling limits.

---

# Pipeline Architecture

* single url vs dual rx/tx vs sync;
* channel_sync accuracy and fallbacks;
* diarization vs dual-track (when to use which);
* spelling and speaker_roles extensibility;
* temp file lifecycle.

---

# API & Integration Architecture

* polling model vs webhooks (future);
* dedup_key for n8n;
* by-dedup vs by job_id;
* client_hint quality;
* auth model (single bearer token).

---

# Docker & Deployment

* single image two roles — pros/cons;
* GPU binding per worker;
* config volumes hot-reload;
* logs persistence;
* network DNS for external audio URLs.

---

# Security Architecture

* API token;
* HF token for pyannote;
* untrusted audio URLs (SSRF risk?);
* secrets in env;
* log redaction.

# Observability

* health endpoint limitations;
* job progress granularity;
* errors.log / crash logs;
* metrics gap (Prometheus, queue depth).
